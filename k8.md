# What Is the Mini Kubernetes Deployment Project?

Kubernetes is used for automating the deployment, scaling, and management of containerized applications. In simple terms, Docker runs containers, while Kubernetes manages containers in a more production-style way. ([Kubernetes][1])

Your project will take the same app from your Docker Compose lab and deploy it using Kubernetes concepts like:

```text
Pods
Deployments
Services
ConfigMaps
Secrets
Ingress
Persistent Volumes
Namespaces
```

The big idea is:

> Docker Compose says: “Run these containers together.”
> Kubernetes says: “Keep this system running in the desired state.”

That is the professional jump.

Your old version probably works like this:

```text

  ↓

  ↓

  ↓

  ↓
```

# Important Kubernetes Concepts You Will Learn

## 1. Pod

A Pod is the smallest running unit in Kubernetes.

In your case:

```text
frontend pod
backend pod
postgres pod
```

Usually, you do not create Pods directly. You use Deployments to manage them.

---

## 2. Deployment

A Deployment manages application Pods and keeps them running. Kubernetes documentation explains that a Deployment manages a set of Pods for an application workload and allows declarative updates. ([Kubernetes][3])

Example idea:

```text
I want 2 backend replicas.
```

Kubernetes tries to keep 2 backend Pods running.

If one crashes, Kubernetes recreates it.

That is the power of orchestration.

---

## 3. Service

A Service gives stable networking to Pods.

Why this matters:

Pods can be destroyed and recreated. Their IP addresses can change.

So instead of the frontend talking directly to a Pod IP, it talks to a Service:

```text
frontend → backend-service
backend → postgres-service
```

This is much cleaner.

---

## 4. ConfigMap

A ConfigMap stores non-secret configuration.

Example:

```text
DB_HOST=postgres-service
DB_PORT=5432
APP_ENV=Development
```

Kubernetes documentation explains that ConfigMaps store non-confidential key-value data and help separate environment-specific configuration from container images. ([Kubernetes][4])

Use ConfigMaps for normal settings.

---

## 5. Secret

A Secret stores sensitive data.

Example:

```text
DB_USER
DB_PASSWORD
JWT_SECRET
```

Kubernetes Secrets are intended for confidential data, although the official docs also warn that Secrets need proper handling because they are not automatically a complete security solution by default. ([Kubernetes][5])

For your project, using Secrets shows that you understand good deployment practice.

---

## 6. Ingress

Ingress controls HTTP access from outside the cluster.

Instead of exposing frontend and backend separately, Ingress can route:

```text
/       → frontend-service
/api    → backend-service
```

Kubernetes documentation describes Ingress as an API object that manages external access to services, typically HTTP, and can provide routing rules based on paths or hostnames. ([Kubernetes][6])

This replaces the role that Nginx played in your Docker Compose version.

---

## 7. Persistent Volume Claim

Your database needs storage.

If a PostgreSQL Pod restarts, you do not want the database data to disappear.

So you use:

```text
PersistentVolumeClaim
```

This gives PostgreSQL persistent storage.

This is one of the most important things to show in the project because it proves you understand that databases are stateful.

# Should You Use Minikube or kind?

For your PC, both are possible now because you have **16GB RAM**.

Minikube’s official start guide lists the basic requirements as 2 CPUs, 2GB free memory, 20GB free disk space, internet, and a container or VM manager. ([minikube][7])

kind runs local Kubernetes clusters using Docker container “nodes,” and it is commonly used for local development and CI-style testing. ([Kind][8])


```text

```

For your portfolio project:

```text

```

